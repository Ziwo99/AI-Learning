Les tags

Les tags sont des références statiques qui pointent vers des versions spécifiques de votre dépôt Git. Ils sont utilisés pour marquer des points importants dans l'historique du projet, tels que les versions stables ou les versions de publication.

Pour créer un tag, vous pouvez utiliser la commande suivante :

git tag <tag_name>

Par exemple, pour créer un tag pour la version 1.0 de votre projet, vous pouvez utiliser la commande :

git tag v1.0

Les tags peuvent également être annotés, ce qui signifie qu'ils peuvent contenir des informations supplémentaires telles que le nom de l'auteur et un message de description. Pour créer un tag annoté, vous pouvez utiliser la commande suivante :

git tag -a <tag_name> -m "Description du tag"

Par exemple, pour créer un tag annoté pour la version 1.0 avec une description, vous pouvez utiliser la commande :

git tag -a v1.0 -m "Version 1.0 - Release"

Les tags peuvent être poussés vers un dépôt distant à l'aide de la commande :

git push origin <tag_name>

Par exemple, pour pousser le tag v1.0 vers le dépôt distant, vous pouvez utiliser la commande :

git push origin v1.0

Les sous-modules

Les sous-modules sont utilisés pour inclure un autre dépôt Git en tant que sous-dossier de votre dépôt principal. Cela permet d'inclure des bibliothèques ou des modules externes dans votre projet sans avoir à les copier manuellement.

Pour ajouter un sous-module à votre dépôt, vous pouvez utiliser la commande suivante :

git submodule add <repository_url> <path_to_submodule>

Par exemple, pour ajouter un sous-module à votre projet à partir du dépôt "https://github.com/example/submodule.git" dans le dossier "submodule", vous pouvez utiliser la commande :

git submodule add https://github.com/example/submodule.git submodule

Après avoir ajouté un sous-module, vous devez l'initialiser et le mettre à jour en utilisant les commandes suivantes :

git submodule init
git submodule update

Cela récupérera les fichiers du sous-module et les placera dans le dossier spécifié.

Les hooks Git

Les hooks Git sont des scripts personnalisés qui sont exécutés automatiquement à des moments spécifiques dans le cycle de vie de Git, tels que la validation d'un commit ou la réception de nouvelles modifications d'un dépôt distant.

Les hooks Git sont stockés dans le dossier ".git/hooks" de votre dépôt et peuvent être des scripts shell, des scripts Python ou tout autre type de script exécutable.

Voici quelques exemples de hooks Git couramment utilisés :

  • pre-commit : exécuté avant la validation d'un commit. Il peut être utilisé pour effectuer des vérifications de style, des tests unitaires ou d'autres tâches de pré-validation.
  • post-commit : exécuté après la validation d'un commit. Il peut être utilisé pour envoyer des notifications, générer de la documentation ou effectuer d'autres tâches post-validation.
  • pre-push : exécuté avant de pousser des modifications vers un dépôt distant. Il peut être utilisé pour effectuer des vérifications supplémentaires avant de partager vos modifications.

Pour activer un hook Git, vous devez le rendre exécutable en utilisant la commande suivante :

chmod +x <hook_name>

Par exemple, pour rendre le hook "pre-commit" exécutable, vous pouvez utiliser la commande :

chmod +x .git/hooks/pre-commit

Les alias Git

Les alias Git sont des raccourcis personnalisés pour les commandes Git couramment utilisées. Ils permettent d'économiser du temps et de la frappe en définissant des commandes plus courtes et plus faciles à retenir.

Pour créer un alias Git, vous pouvez utiliser la commande suivante :

git config --global alias.<alias_name> <command>

Par exemple, pour créer un alias "co" pour la commande "checkout", vous pouvez utiliser la commande :

git config --global alias.co checkout

Une fois l'alias créé, vous pouvez l'utiliser comme une commande Git normale. Par exemple, au lieu de taper "git checkout", vous pouvez simplement taper "git co".

Les fichiers .gitignore

Les fichiers .gitignore sont utilisés pour spécifier les fichiers et les dossiers que Git doit ignorer lors de l'ajout et de la validation des modifications. Cela permet d'éviter d'inclure des fichiers temporaires, des fichiers de configuration sensibles ou d'autres fichiers non pertinents dans votre dépôt Git.

Pour créer un fichier .gitignore, vous pouvez simplement créer un fichier nommé ".gitignore" à la racine de votre dépôt et y ajouter les règles d'ignorance.

Voici quelques exemples de règles couramment utilisées dans un fichier .gitignore :

# Ignorer tous les fichiers .txt
*.txt

# Ignorer le dossier "build"
/build

# Ignorer les fichiers de configuration sensibles
config.ini

# Ignorer les fichiers de sauvegarde
*.bak

Les règles dans un fichier .gitignore peuvent utiliser des caractères génériques tels que "*" pour représenter plusieurs caractères ou "?" pour représenter un seul caractère.

Il est important de noter que les fichiers déjà suivis par Git ne seront pas ignorés même s'ils correspondent à une règle dans le fichier .gitignore. Pour ignorer ces fichiers, vous devrez les supprimer de l'historique Git à l'aide de la commande "git rm --cached".
