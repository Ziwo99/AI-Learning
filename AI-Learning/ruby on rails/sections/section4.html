Introduction aux tests

Les tests sont une partie essentielle du développement d'applications Ruby on Rails. Ils permettent de vérifier que le code fonctionne correctement et de détecter les éventuels problèmes ou bugs. Les tests garantissent également que les nouvelles fonctionnalités ajoutées à l'application ne cassent pas les fonctionnalités existantes.

Les tests en Ruby on Rails sont généralement écrits à l'aide du framework de tests intégré appelé "RSpec". RSpec fournit une syntaxe expressive et facile à lire pour écrire des tests.

Les tests unitaires

Les tests unitaires sont utilisés pour tester des parties spécifiques du code, appelées "unités". Une unité peut être une méthode, une classe ou même un module. Les tests unitaires vérifient que chaque unité fonctionne correctement de manière isolée.

Voici un exemple de test unitaire en Ruby on Rails :

require 'rails_helper'

RSpec.describe User, type: :model do
  it "is valid with valid attributes" do
    user = User.new(name: "John Doe", email: "john@example.com")
    expect(user).to be_valid
  end

  it "is not valid without a name" do
    user = User.new(email: "john@example.com")
    expect(user).to_not be_valid
  end

  it "is not valid without an email" do
    user = User.new(name: "John Doe")
    expect(user).to_not be_valid
  end
end

Dans cet exemple, nous testons le modèle "User". Nous vérifions qu'un utilisateur est valide avec des attributs valides, et qu'il n'est pas valide sans nom ou sans email.

Les tests d'intégration

Les tests d'intégration sont utilisés pour tester le fonctionnement de l'application dans son ensemble. Ils vérifient que les différentes parties de l'application fonctionnent correctement ensemble.

Voici un exemple de test d'intégration en Ruby on Rails :

require 'rails_helper'

RSpec.describe "Users", type: :request do
  describe "GET /users" do
    it "returns a 200 OK status" do
      get "/users"
      expect(response).to have_http_status(200)
    end
  end
end

Dans cet exemple, nous testons la requête GET "/users". Nous vérifions que la réponse renvoie un statut 200 OK.

Les tests fonctionnels

Les tests fonctionnels sont utilisés pour tester le comportement des contrôleurs de l'application. Ils vérifient que les actions des contrôleurs fonctionnent correctement et renvoient les bonnes réponses.

Voici un exemple de test fonctionnel en Ruby on Rails :

require 'rails_helper'

RSpec.describe UsersController, type: :controller do
  describe "GET #index" do
    it "returns a success response" do
      get :index
      expect(response).to be_successful
    end
  end
end

Dans cet exemple, nous testons l'action "index" du contrôleur "UsersController". Nous vérifions que la réponse est un succès.

Les tests de performance

Les tests de performance sont utilisés pour évaluer les performances de l'application. Ils permettent de détecter les éventuels problèmes de performance et d'optimiser le code si nécessaire.

Voici un exemple de test de performance en Ruby on Rails :

require 'rails_helper'
require 'benchmark'

RSpec.describe "Performance", type: :performance do
  it "performs the task in less than 1 second" do
    expect {
      Benchmark.realtime do
        # Code à tester
      end
    }.to be < 1.0
  end
end

Dans cet exemple, nous utilisons la classe Benchmark pour mesurer le temps d'exécution d'une tâche. Nous vérifions que le temps d'exécution est inférieur à 1 seconde.

Les tests de sécurité

Les tests de sécurité sont utilisés pour vérifier que l'application est sécurisée contre les attaques courantes, telles que les injections SQL ou les attaques XSS.

Voici un exemple de test de sécurité en Ruby on Rails :

require 'rails_helper'

RSpec.describe "Security", type: :request do
  it "prevents SQL injection" do
    user = User.create(name: "John Doe", email: "john@example.com")
    malicious_input = "'; DROP TABLE users; --"
    post "/users", params: { name: malicious_input, email: "test@example.com" }
    expect(User.find_by(name: malicious_input)).to be_nil
  end
end

Dans cet exemple, nous testons si l'application est vulnérable aux injections SQL. Nous créons un utilisateur avec un nom malveillant contenant une instruction SQL, puis nous vérifions que l'utilisateur n'est pas enregistré dans la base de données.
